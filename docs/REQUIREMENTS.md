# Gruntia 需求文档

版本：0.2  
日期：2026-04-24  
项目定位：轻量级 2D 俯视角基地经营 + 卡牌化探索 Demo

## 1. 项目概述

Gruntia 是一款轻量级 2D 俯视角游戏 Demo，融合基地建设、生存经营、RPG 势力成长、Roguelike 随机探索与卡牌回合制行动系统。项目目标不是制作完整商业游戏，而是先构建一个可双击运行、核心循环明确、可继续扩展的原型产品。

游戏的一局流程不是短时间通关式冒险，而是围绕玩家势力进行长期推进。玩家在固定基地中建设、生产、维持生存、武装角色、解锁科技；随后派遣角色进入随机生成的野外地图探索。野外探索阶段中，除撤离操作外，移动、采集、战斗、互动等行动都通过回合制卡牌消耗完成。玩家可以随时存档、读档；当玩家势力下的所有角色死亡时，本局游戏失败。

## 2. 产品目标

1. 提供一个可直接运行的 Windows 桌面游戏 Demo。
2. 不依赖 Unity、Godot、Unreal 等完整游戏引擎。
3. 用户无需安装运行环境，双击启动文件即可游玩。
4. 用尽量小的技术栈实现清晰、可维护、可扩展的 2D 游戏框架。
5. 验证以下核心玩法能否组合成一个轻量但完整的长期循环：
   - 基地建设与生存经营
   - 角色武装、势力成长与科技解锁
   - 随机野外地图探索
   - 基于卡牌的回合制行动系统
   - 卡牌回合制战斗

## 3. 目标平台

### 3.1 必须支持

- Windows 10/11 x64。
- 用户通过 `Gruntia.exe` 或发布包中的启动文件直接启动游戏。
- 发布包内包含所有必要依赖、资源与配置文件。

### 3.2 尽量支持

- macOS Apple Silicon / Intel。
- macOS 支持不作为首个 Demo 的硬性验收条件，但技术选型应避免明显锁死 Windows。

## 4. 技术路线

### 4.1 推荐技术栈

首选方案：Rust + 轻量窗口/渲染/音频库。

建议模块：

- 语言：Rust。
- 窗口与输入：`winit`。
- 2D 像素渲染：`pixels` 或基于 `wgpu` 的轻量封装。
- 音频：`rodio`。
- 存档与配置：`serde` + `ron` / `json`。
- 随机数：`rand`。
- 日志：`tracing` 或 `log`。

选择理由：

- 可编译为原生可执行文件，适合免安装发布。
- 不使用完整游戏引擎，保留自研游戏循环、实体系统、地图系统、卡牌行动系统和存档系统。
- Windows 与 macOS 都有较好支持。
- 依赖体积可控，适合轻量 Demo。

### 4.2 不采用内容

- 不使用 Unity、Godot、Unreal、GameMaker 等完整游戏引擎。
- 不要求浏览器运行，不以 Web 游戏为首要形态。
- 不要求联网、账号、云存档、多人联机。
- 不要求首版支持手柄。

## 5. 游戏核心体验

### 5.1 玩家视角

- 2D 俯视角。
- 基地地图与野外探索地图均以瓦片或近似格子的方式呈现。
- 玩家控制的不是单一英雄，而是一个可成长的生存势力。
- 势力中可以拥有多个角色；角色可被派遣到野外探索，也可能在探索或战斗中死亡。

### 5.2 核心循环

整局游戏由无限重复的两阶段循环构成：

1. 基地建设阶段。
2. 野外探索阶段。
3. 撤离后返回基地，进入下一轮基地建设。

只要势力仍有至少一名存活角色，游戏就可以继续推进。探索失败可能造成派遣角色死亡、资源损失或探索收益丢失，但不必然导致整局失败。

### 5.3 基地建设阶段

基地建设阶段是玩家的长期经营核心。

玩家在此阶段可以：

- 建设和升级基地建筑。
- 维持角色生存需求，例如食物、治疗、休息。
- 制作装备、道具或探索补给。
- 武装角色并编组探索队伍。
- 解锁科技，开放新建筑、新卡牌、新装备、新基地面积或新探索区域。
- 整理背包、仓库、牌组和角色状态。
- 存档、读档或退出游戏。

基地阶段原则上不使用卡牌行动限制，操作应偏经营和规划。

### 5.4 野外探索阶段

野外探索阶段发生在随机生成的探索地图中。

探索阶段中，除撤离操作外，玩家的所有行动都需要通过回合制卡牌系统执行，包括：

- 移动。
- 采集。
- 搜索。
- 战斗攻击。
- 防御。
- 使用道具。
- 互动事件。
- 建立临时设施。
- 拆除或清理障碍。

探索阶段强调资源压力、手牌选择和风险管理。玩家需要在有限手牌、行动点或能量约束下决定是否深入、战斗、采集或撤离。

### 5.5 失败条件

整局游戏的唯一核心失败条件：

- 玩家势力下的所有角色死亡。

单次探索队伍全灭不一定等于整局失败；如果基地中仍有其他存活角色，玩家可以继续游戏。若最后一名角色死亡，则本局失败并进入失败结算。

### 5.6 Demo 阶段目标

首版 Demo 不要求无限内容量，但需要表现长期循环雏形：

- 建立一个固定基地。
- 派遣角色进入至少一种随机探索地图。
- 在探索中通过卡牌完成移动、采集和战斗。
- 撤离后把资源带回基地。
- 用资源建造或升级至少一种基地设施。
- 解锁至少一项科技或新内容。
- 支持随时存档和读档。

## 6. 地图系统

### 6.1 基地地图

基地地图是固定的正方形空地。

必须包含：

- 基地初始尺寸固定，例如 16x16 或 24x24。
- 地图以瓦片网格管理。
- 建筑只能放置在合法空地上。
- 基地地图不会随机重置。
- 基地状态进入存档。

科技发展可以逐步扩大基地面积。

扩展方式建议：

- 初始基地为中心正方形区域。
- 科技解锁后向外扩展一圈或开放相邻区块。
- 扩展后的空地可用于更多建筑、道路或防御设施。

### 6.2 探索地图

探索地图随机生成。

必须包含：

- 随机地形布局。
- 随机资源点。
- 随机敌人。
- 随机事件或可交互对象。
- 撤离点或随时撤离机制。

首版建议：

- 地图尺寸控制在 32x32 到 64x64。
- 使用随机种子，便于调试和复现。
- 先做轻量随机组合，不做大型复杂地形生成。

## 7. 核心系统需求

### 7.1 基地建设系统

必须包含：

- 建筑放置、拆除或升级。
- 建筑消耗资源。
- 建筑影响生产、存储、治疗、科技或探索准备。
- 建筑状态保存到存档。

首版建筑建议：

- 篝火：提供基础恢复和营地中心。
- 仓库：提高资源存储上限。
- 工作台：制作基础装备或工具。
- 研究台：解锁科技。
- 训练桩：提升角色基础战斗能力或解锁战斗卡。

### 7.2 生存资源系统

必须包含至少 3 种基础资源：

- 木材。
- 石料。
- 食物。

资源来源：

- 野外探索采集。
- 野外事件奖励。
- 战斗奖励。
- 基地建筑生产，首版可选。

资源用途：

- 建造和升级建筑。
- 制作装备、工具或补给。
- 维持角色生存。
- 解锁科技。

### 7.3 角色与势力系统

必须包含：

- 势力拥有一个或多个角色。
- 角色具有生命值、攻击、防御、行动能力等基础属性。
- 角色可以死亡。
- 角色死亡后从势力中移除或标记为死亡。
- 所有角色死亡时游戏失败。

首版建议：

- 初始提供 2 到 3 名角色，降低单次探索死亡带来的挫败感。
- 每次探索最多派遣 1 到 3 名角色。
- 角色成长以局内永久成长为主，绑定当前存档。

### 7.4 科技系统

必须包含：

- 科技通过基地建筑或资源解锁。
- 科技可以开放新建筑、新卡牌、新装备、新地图区域或基地扩建。
- 科技状态进入存档。

首版科技建议：

- 基地扩建 I：扩大基地正方形区域。
- 简易工具：解锁更高效采集卡。
- 木工：解锁木墙或仓库升级。
- 战斗训练：解锁基础攻击卡或防御卡。

### 7.5 野外卡牌行动系统

这是探索阶段的核心系统。

必须包含：

- 探索阶段按回合推进。
- 每回合抽取手牌。
- 玩家使用卡牌执行行动。
- 卡牌消耗能量、行动点或其他资源。
- 除撤离操作外，探索阶段所有主动行动都由卡牌触发。
- 回合结束后敌人、环境或事件可以行动。

首版卡牌类型：

- 移动牌：移动指定距离或方向。
- 采集牌：采集相邻或指定范围资源。
- 搜索牌：揭示周围区域、发现隐藏事件或提高收益。
- 攻击牌：对敌人造成伤害。
- 防御牌：获得格挡或减伤。
- 技能牌：抽牌、恢复、强化或施加状态。
- 工具牌：消耗工具或补给产生特殊效果。

### 7.6 战斗系统

战斗不应完全脱离探索，而是探索卡牌行动系统的一种高风险状态。

必须包含：

- 野外探索中遭遇敌人后进入战斗状态或战斗区域。
- 战斗仍使用卡牌和回合制规则。
- 玩家通过攻击、防御、技能、移动等卡牌处理敌人。
- 敌人有明确行动意图，例如攻击、防御、追击、蓄力。
- 玩家角色或敌人生命值归零后结算。

首版建议：

- 战斗可在探索地图原地发生，不强制切入完全独立界面。
- 若实现成本更低，也可以先切换到独立战斗界面，但必须保持“卡牌行动”规则一致。

### 7.7 Roguelike 随机探索系统

必须包含：

- 探索地图随机生成。
- 资源、敌人和事件随机分布。
- 探索收益存在随机性。
- 探索具有撤离决策。
- 深入探索带来更高收益和更高风险。

注意：

- 本项目不是传统“一死重开”的短局 Roguelike。
- Roguelike 元素主要体现在野外探索的随机性、风险收益、遭遇事件和卡牌奖励。

### 7.8 存档系统

必须支持随时存档、读档。

存档应至少包含：

- 基地地图与建筑状态。
- 势力资源。
- 角色列表、属性、生命状态和装备。
- 科技解锁状态。
- 当前牌组、已解锁卡牌和探索卡牌状态。
- 当前所在阶段：基地或探索。
- 如果在探索中存档，需要保存探索地图、敌人、事件、手牌、抽牌堆、弃牌堆、当前回合等状态。
- 游戏设置。

首版可以只提供一个自动存档槽和一个手动存档槽，但读档流程必须稳定。

## 8. UI 与交互需求

### 8.1 主菜单

必须包含：

- 新游戏。
- 继续游戏。
- 读取存档。
- 设置。
- 退出。

### 8.2 基地 UI

必须显示：

- 当前资源数量。
- 当前角色与状态。
- 建筑菜单。
- 科技入口。
- 探索出发入口。
- 存档与读档入口。

### 8.3 探索 UI

必须显示：

- 探索地图。
- 当前队伍角色状态。
- 手牌区域。
- 抽牌堆、弃牌堆数量。
- 当前能量或行动点。
- 当前回合。
- 撤离按钮。
- 当前格或目标对象的交互信息。

### 8.4 战斗 UI

必须显示：

- 玩家角色生命、格挡、状态。
- 敌人生命、状态、行动意图。
- 手牌区域。
- 抽牌堆、弃牌堆数量。
- 当前能量或行动点。
- 结束回合按钮。

### 8.5 设置界面

首版必须包含：

- 音量开关或音量滑块。
- 窗口模式/全屏切换。
- 返回主菜单。

## 9. 美术与音频需求

### 9.1 美术风格

首版建议使用低成本像素风或简洁色块风。

要求：

- 角色、敌人、资源、建筑物需要有清晰辨识度。
- 基地地图和探索地图应有明显视觉差异。
- 卡牌 UI 需要可读性优先。
- 地图信息可读性优先于装饰性。

### 9.2 音频

首版建议包含：

- 按钮点击音效。
- 建造音效。
- 科技解锁音效。
- 抽牌/出牌音效。
- 移动音效。
- 采集音效。
- 受击音效。
- 撤离成功音效。
- 胜利/失败音效。

背景音乐可选。

## 10. 数据与配置

游戏内容应尽量数据驱动，方便后续扩展。

建议配置文件：

- `cards.json`：卡牌定义。
- `characters.json`：角色模板或初始角色定义。
- `enemies.json`：敌人定义。
- `buildings.json`：建筑定义。
- `technologies.json`：科技定义。
- `items.json`：资源、装备与物品定义。
- `maps.json`：地图生成参数。
- `settings.json`：用户设置。

首版可先使用 JSON，后续如需更强可读性可改为 RON 或 TOML。

## 11. 发布需求

### 11.1 Windows 发布包

发布包结构建议：

```text
Gruntia/
  Gruntia.exe
  assets/
  data/
  saves/
  config/
  README.txt
```

要求：

- 用户解压后双击 `Gruntia.exe` 即可启动。
- 不要求用户安装 Rust、.NET、Python、Node.js 或其他运行环境。
- 如果存在动态库，必须随发布包一并提供。
- 存档目录应位于发布包内或用户数据目录中，具体位置需要在 README 中说明。

### 11.2 macOS 发布包

可选目标：

- 提供 `.app` 或可执行文件加资源目录。
- 不作为首版验收阻塞项。

## 12. MVP 范围

### 12.1 必须完成

- 原生窗口启动。
- 主菜单。
- 固定正方形基地地图。
- 基地建筑放置与基础升级。
- 至少 3 种资源。
- 至少 4 种基地建筑。
- 至少 1 项基地扩建科技。
- 至少 2 名可用角色。
- 随机生成的野外探索地图。
- 探索阶段回合制卡牌行动。
- 移动、采集、战斗均通过卡牌执行。
- 撤离操作。
- 至少 3 种敌人。
- 至少 20 张卡牌，其中必须包含移动、采集、攻击、防御、搜索类型。
- 随时存档、读档。
- 势力全灭失败结算。
- Windows 可双击运行发布包。

### 12.2 可以暂缓

- 完整剧情。
- 大型地图。
- 复杂装备词条。
- 复杂制作链。
- 多基地。
- 多层地牢。
- 敌对势力 AI。
- 联机。

- Mod 支持。

## 13. 验收标准

项目达到首版 Demo 可验收状态时，应满足：

1. Windows 电脑上解压发布包后可直接双击启动。
2. 从主菜单可以开始新游戏、继续游戏或读取存档。
3. 玩家可以在固定正方形基地中建设建筑。
4. 玩家可以通过科技扩大基地面积。
5. 玩家可以派遣角色进入随机探索地图。
6. 野外探索中的移动、采集和战斗行动均通过卡牌完成。
7. 玩家可以从探索地图撤离并带回资源。
8. 玩家可以用探索收益反哺基地建设或科技解锁。
9. 游戏支持探索中和基地中的存档、读档。
10. 当势力所有角色死亡时进入失败结算。
11. 游戏崩溃、卡死、无法退出等阻塞问题不存在。
12. README 中包含启动方式、操作方式、存档位置和已知问题。

## 14. 建议开发阶段

### 阶段 1：项目骨架

- 创建 Rust 项目结构。
- 建立窗口、输入、主循环、资源加载。
- 实现基本渲染、UI 框架与调试日志。

### 阶段 2：基地原型

- 实现固定正方形基地地图。
- 实现建筑放置。
- 实现资源库存。
- 实现基地 HUD。

### 阶段 3：角色、科技与存档

- 实现势力角色列表。
- 实现角色生命、死亡和基础属性。
- 实现科技解锁。
- 实现基地扩建。
- 实现随时存档、读档。

### 阶段 4：探索地图原型

- 实现随机探索地图。
- 实现探索队伍出发与撤离。
- 实现资源点、敌人和事件生成。

### 阶段 5：野外卡牌行动

- 实现探索回合状态机。
- 实现牌组、抽牌、弃牌、能量、出牌。
- 实现移动牌、采集牌、搜索牌。

### 阶段 6：战斗整合

- 实现攻击牌、防御牌、技能牌。
- 实现敌人行动意图和敌人回合。

## 15. 风险与约束

### 15.1 主要风险

- “所有探索行动都卡牌化”会增加交互复杂度，尤其是移动和采集的手感。
- 基地经营、探索地图和战斗系统同时存在，首版范围容易膨胀。
- 随时存档要求保存更多运行时状态，特别是探索中牌堆、手牌、敌人和地图状态。
- 不使用游戏引擎意味着需要自行维护主循环、资源管理、UI、状态切换和存档序列化。

### 15.2 控制策略

- 首版基地小、地图小、卡牌少，优先验证闭环。
- 探索行动统一走同一套卡牌解析流程，避免移动、采集、战斗各自实现一套规则。
- 存档系统早做，避免后期重构大量状态结构。
- 每个阶段都保持可运行状态。
- Demo 首版只做一种探索地貌和少量科技，扩展内容留到后续。

## 16. 当前默认设定

如后续没有额外设定，项目暂按以下方向推进：

- 世界观：荒野遗迹求生。
- 玩家身份：带领小型流浪者营地的首领。
- 核心目标：建设基地、派遣角色探索、带回资源、解锁科技并延续势力。
- 基地形态：固定正方形空地，可通过科技逐步扩大。
- 探索形态：随机生成的野外地图。
- 美术方向：简洁像素风。
- 单次探索时长：5 到 15 分钟。
- 整局时长：可持续推进，直到势力全灭或玩家主动重开。
- 操作方式：键盘鼠标。
- 卡牌形式：探索与战斗共享一套回合制卡牌行动规则。
